---
title: Lane
description: "Agent-native payments. Your agent drafts a purchase, you authorize it, and Lane completes the checkout at the real merchant — credentials and card data never leave the vault."
---

Lane turns an AI agent into a buyer. The agent says what to purchase, where, and
how much; Lane runs the full checkout at the real merchant (find the product,
add to cart, sign in if needed, pay, confirm) and reports back. Raw card numbers
never reach Lane — cards are tokenized, payment credentials are single-use, and
every purchase is bounded by an authorization the buyer sets.

There are two ways to use Lane, built on the same payment rails — one for an
**agent and a person working together**, one for an **agent or app running on its
own**.

<CardGroup cols={2}>
  <Card title="Lane Plugin" icon="terminal" href="/plugin/installation">
    An **interactive** payment surface a person and their agent share. The agent
    drafts a purchase, the person approves it with a **passkey tap**, and Lane
    drives the merchant checkout. Agent-agnostic and general-purpose — point it at
    anything you want to buy. Distributed today as a Claude Code plugin.
  </Card>
  <Card title="Order API" icon="code" href="/api/overview">
    A **programmatic** checkout endpoint that an agent or app calls as a step in a
    workflow. One `POST /orders` with a plain-language instruction and a ship-to
    address; Lane works the order in the background and notifies you by webhook.
    No human in the loop per order — spend is pre-authorized.
  </Card>
</CardGroup>

## Which one do I want?

|                            | Lane Plugin                                     | Order API                                          |
| -------------------------- | ----------------------------------------------- | -------------------------------------------------- |
| **Who interacts with it**  | An agent **and** an end-user, together          | An agent or app, on its own, inside a workflow     |
| **Human in the loop**      | Yes — the person approves every purchase        | No per-order approval — spend is pre-authorized    |
| **Approval**               | Passkey tap in the browser, each time           | A one-time payment method + spending authorization |
| **Surface**                | The agent's environment (Claude Code today)     | HTTPS API + webhooks                               |
| **Latency model**          | Conversational, live                            | Asynchronous (an order takes a few minutes)        |
| **Best for**               | Open-ended, hands-on purchasing with a person   | Embedding checkout into an automated workflow at scale |

Both share one model — purchases are organized as an **intent** (the
authorization) and its **mandates** (each a single merchant checkout, an
"order"). The [shared concepts page](/concepts/model) explains it once; you don't
need it to get started.
